# Renovation Planning Assistant

Хакатонный MVP для анализа плана квартиры, AI-консультации по перепланировке и визуализации результата в 2D/3D.

Проект объединяет чат-интерфейс, распознавание floor plan из изображения, локальную LLM-проверку и отдельный сервис хранения планов. Это репозиторий, который можно показать в резюме: здесь есть понятная архитектура, рабочий пользовательский сценарий и живая full-stack интеграция.

## Что делает проект

- принимает изображение плана помещения;
- преобразует его в структурированный JSON;
- позволяет открыть результат в 2D/3D-редакторе;
- даёт AI-ответы по перепланировке через локальную LLM;
- сохраняет планы в PostgreSQL.

## Почему проект интересный

- Это не просто UI-макет: в репозитории есть frontend, API-слой и отдельный storage-service.
- Есть реальный пользовательский сценарий от загрузки плана до визуализации и сохранения.
- В проекте сочетаются React, Spring Boot, PostgreSQL, 3D-визуализация и AI-интеграция.

## Архитектура

Репозиторий состоит из трёх частей:

- `frontend` - React-приложение с чатом, загрузкой изображения и 2D/3D-сценой;
- `api-service` - Spring Boot API для LLM, парсинга плана и маршрутизации запросов;
- `plan-service` - Spring Boot сервис хранения планов в PostgreSQL.

Поток данных выглядит так:

1. Пользователь загружает изображение плана.
2. Фронтенд отправляет изображение на сервис распознавания.
3. Полученный JSON открывается в редакторе.
4. API может отправить план в локальную LLM для анализа и рекомендаций.
5. План сохраняется в PostgreSQL через `plan-service`.

## Tech Stack

- Frontend: React, React Router, Three.js, React Three Fiber
- Backend: Java 21, Spring Boot
- Storage: PostgreSQL
- AI: Ollama + `llama3`
- Infra: Docker Compose

## Основные возможности

- чат-интерфейс для взаимодействия с AI;
- загрузка изображения floor plan;
- 2D/3D просмотр помещения;
- размещение мебели в сцене;
- сохранение и получение планов через REST API;
- Swagger-документация для backend API.

## Быстрый запуск

### 1. PostgreSQL

```bash
cd plan-service
docker compose up -d
```

### 2. Plan Service

```bash
cd plan-service
./mvnw spring-boot:run
```

### 3. Ollama

Проект ожидает локальную модель по адресу:

```text
http://localhost:11434/api/generate
```

Пример запуска:

```bash
ollama run llama3
```

### 4. API Service

```bash
cd api-service
./mvnw spring-boot:run
```

По умолчанию сервис стартует в профиле `demo`, поэтому парсинг плана может работать через локальный mock-файл.

### 5. Frontend

```bash
cd frontend
npm install
npm start
```

Приложение будет доступно на `http://localhost:3000`.

## API

`api-service`:

- `POST /api/parse` - разбор изображения плана;
- `POST /api/validate` - AI-проверка плана;
- `POST /api/llm` - чат с локальной LLM;
- `GET /api/plans` - список сохранённых планов.

Swagger UI:

```text
http://localhost:8080/swagger-ui.html
```

`plan-service`:

- `GET /api/plans` - постраничный список планов;
- `POST /api/plans` - сохранение плана.

## Что можно улучшить

Как и у большинства хакатонных проектов, здесь есть точки роста:

- вынести секреты и API keys из клиентского кода;
- унифицировать интеграцию распознавания плана;
- добавить миграции БД;
- собрать единый `docker-compose` для всего стека;
- покрыть ключевые сценарии тестами.
